Q: What is Same origin Policy ?
 * The same origin(same site) policy limits the access of windows and frames to each other.
 * Suppose a user has 2 tabs open. 1 is fb.com and other is sbi.com . Then fb.com cannot access
  a script from sbi.com.
 * So the purpose of "Same Origin" policy is to protect users from information theft.

Q: How can we say that 2 urls have same origin ?
 * Two urls are said to have "same origin" if they have the same 
   -> protocol
   -> domain
   -> port.

Q: Explain postMessage() ?
 * The postMessage() method allows windows to talk to each other no matter which origin they are from.
 * The window.postMessage() method safely enables cross-origin communication between Window object.
   e.g between a page and a pop-up or between a page and an iframe embedded within it.

Q: Whta is abort controller ?
 * As we know fetch returns a promise. JS has generally no concept of aborting a promise.
 * AbortController is a special built-in object that is used to abort not only fetch, but other asynchronous tasks as well.
 * AbortController is a simple object that generates an abort event on its siginal property when the abort() 
   method is called (and also siginal.aborted to true)

Q: What is CORS?
 * CORS stands fro Cross Origin resource Sharing
 * Cross Origin requests are those sent to another domain or port or protocol. It requires a special headers from remote site.
 * That policy is called CORS.
 * If a request is cross-origin browser always adds an Origin header to it.
 * CORS exists to protect the internet from evil hackers
 * CORS are of 2 types : 
      1- Safe requests
      2- All the others

Q: Explain Safe Request in CORS ?
 * A request is safe if it satisfies 2 conditions :
   1- Safe method: GET, POST or HEAD
   2- Safe Headers: the only allowed custom headers are:
      * Accept
      * Accept-language
      * Content-language
      * Content-type - (application/x-www-form-urlencoded, multipart/form-data or text/plain)
 * Any other requests are considered as unsafe. i.e PUT method or with API-key HTTP-headers 
   does not fit the limitations

Q: Explain Unsafe Request in CORS ?
 * When we try to make a unsafe request, the browser sends a special "preflight" request that 
   asks the server - does it aceept such cross-origin requests, or not.
 * The browser sends an OPTIONS request to the same URL, with the headers.
   * Access-Control-Request-Method :  has requested method
   * Access-Control-Request-Headers: lists unsasfe requested headers
 * The server should respond with status 200 and the headers:
   * Access-Control-Allow-Methods: with a list of allowed methods.
   * Access-Control-Allow-Headers: with a list of allowed headers.
   * Access-Control-Max-Age: with the number of seconds to cache the permissions
 * Then the actual request is sent, and the previous "safe" scheme is applied.

Q: What is long polling ?
 * Long polling is the simplest way of having persistent connection with the server, that doesn't
   use any specific protocol like WebSocket or Server Side Events.
 * The flow of long polling :
   * A request is sent to the server
   * The server doesn't close the connection untill it has a message to send.
   * When a message appears - the server responds to the request with it.
   * The browser makes a new request immediately.
 * The situation when the browser sent a request and has a pending connection with the the server, is
     starndard for this method. Only when a message is delivered, the connection is reestablished.
 * If the connection is lost, because of network issue, the browser immediately sends a new request.
 * Long polling works great in situation when messages are rare.
 * If messages come very often, use WebSocket or Server Sent Events.

Q: What is WebSocket ?
 * WebSocket allows two-way, event-driven communication between web browser and server.
 * WebSocket protocol provides a way to exchange data between browser and server via persistent connection.
 * The data can be passed in both directions as "packets", without breaking the connection and additional 
   HTTP-requests.
 * Web Socket is a great way for serving continuous data exchange, e.g online games, real-time chat application
 * Once socket is created, we should listen to events on it. There are 4 events :
    1: Open - Connection established
    2: message - data received
    3: error - websocket error
    4: close - connection closed

 * Server sent events is a pushing technology that enables pushing notification/message/events from the

Q: What are Cookies?
 * Cookies are small strings of data that are stored directly in the browser. They are a part of the HTTP protocol.
 * Cookie are usually set by a web-server using the response Set-cookie HTTP-header.
 * Then the browser add them to almost every request to the same domain using the Cookie HTTP-header.
 * We can also access cookie from document.cookie property

Q: Usecase of Cookie ?
 * One of the most widespread usecase of cookie is authentication.
  1- Once signed in, the server uses Set-Cookie HTTP-header in the response to set 
      a cookie with unique "session identifier".
  2- Next time when the request is sent from the same domain, the browser sends the cookie over the network
      using the Cookie HTTP-header.
  3- So the server knows who made the request.

Q: What are session cookies ?
 * By default a cookie doesn't have any options or attributes, it disappears when the browser is closed.
    Such cookies are called 'Session Cookies'.
 * If we want cookie to survive browser close, we can set the expires or max-age option.

Q: What is XSRF ?
 * Cross-Site Request Forgery (CSRF) is an attack that forces an end user to execute unwanted actions on 
    a web application in which they’re currently authenticated. 
 * With a little help of social engineering (such as sending a link via email or chat), an attacker may 
    trick the users of a web application into executing actions of the attacker’s choosing. 
 * If the victim is a normal user, a successful CSRF attack can force the user to perform state changing 
    requests like transferring funds, changing their email address, and so forth. 
 * If the victim is an administrative account, CSRF can compromise the entire web application.

Q: Example of CSRF ? 
 * Imagine you have logged in to bank.com, there you are authenticated with a token.
 * Your browser send that cookie to bank.com with every request, so bank.com recognizes you and you perform
    all sensitive financial operations.
 * Now while browsing the web in another window, you accidentally come to evil.com.
 * evil.com has a JS code that initiates a payment transaction from bank.com to the hacker account.
 * The browser send cookie every time you visit the site bank.com, even if the payment form was submitted 
    from evil.com. So the bank thinks its you and performs the transaction.

Q: How to prevent XSRF ?
 * The cookie samesite option provides a way to protect from XSRF attack.
 * use samesite=strict

Q: What is third-party cookie ?
 * A cookie is called third-party if its placed by a domain other than the page the user is visiting.
